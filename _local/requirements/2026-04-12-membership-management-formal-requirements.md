# FlowTennis 订场会员管理正式需求方案

> 本文档是后续会员模块开发的正式需求基线。  
> 目标不是重造一套新会员系统，而是在现有真实业务和现有代码模型之上，完成一次“结构清晰、交互简单、规则一致”的重构。  
> 如本文件与 Claude 原型稿、临时讨论口径冲突，以本文件为准。  
> 如本文件与既有代码实现冲突，一期开发以“少改底层、优先统一交互与口径”为原则，二期再处理模型升级。

---

## 1. 文档定位

### 1.1 本文档解决什么问题

当前会员相关问题，不是单点 UI 难看，而是以下 4 类内容混在一起：

1. 订场用户 CRM 信息
2. 会员账户当前状态
3. 会员购买成交记录
4. 赠送权益批次与权益流水

现状结果：

1. 管理员在“编辑订场用户”时，同时看到资料、财务、会员、权益操作，认知负担过高。
2. 页面按“数据库表”组织，而不是按“管理员动作”组织。
3. 一些业务规则虽然已经写进后端，但前端没有讲清楚，导致管理员不敢操作。
4. 讨论稿里出现了和现有代码不一致的规则口径，后续如果不统一，开发一定返工。

### 1.2 本文档用于什么

本文件用于指导：

1. 会员页面重构
2. 管理后台会员操作交互重构
3. 会员与订场用户、学员、订场财务之间的边界统一
4. 一期开发范围划定

### 1.3 本文档不负责什么

本文件不负责：

1. C 端会员展示
2. 财务退款审批流程
3. 家庭共享会员
4. 会员与课包权益合并
5. 教练端会员展示

## 2. 现有系统真实基线

本节不是理想设计，而是当前系统已经存在、开发必须尊重的真实基础。

### 2.1 当前真实数据对象

当前代码中，会员相关对象已经拆分为独立表：

1. `ft_membership_plans`
2. `ft_membership_accounts`
3. `ft_membership_orders`
4. `ft_membership_benefit_ledger`
5. `ft_membership_account_events`

余额主账不在会员表中，而在：

1. `ft_courts.history`

这意味着：

1. 会员不是独立客户体系，而是挂在订场用户上的一套会员状态。
2. 一个订场用户只有一个“当前会员账户视图”。
3. 一个订场用户可以有多条会员购买记录。
4. 当前代码里没有单独的“权益批次表”，所谓“权益批次”是由购买记录和权益流水推导出来的展示概念。

### 2.2 当前真实业务关系

1. 订场用户是会员主体。
2. 学员只做关联展示，不是会员主对象。
3. 会员充值会写入 `courts.history`，形成余额增加。
4. 订场消费如果使用储值扣款，会从余额中扣减。
5. 折扣影响的是订场消费扣款金额，不直接修改余额数值。
6. 权益消耗不影响余额。

### 2.3 当前真实后端规则口径

当前代码和测试已经体现的真实规则如下：

1. 会员方案默认 `validMonths = 12`
2. 会员方案默认 `maxMonths = 24`
3. 购买后：
   - `validUntil = purchaseDate + 12个月`
   - `hardExpireAt = purchaseDate + 24个月`
4. 如果仍在有效期内，并且本次充值金额不低于上一次“合规续充金额”，才重置有效期。
5. 如果低于上一次合规档位，允许充值，但不重置有效期。
6. 权益有效期跟随当次购买记录，默认 `purchaseDate + 12个月`
7. 权益消耗按最早到期批次优先扣减。
8. 到 `validUntil` 后若仍有余额，可自动进入 `extended`
9. 到 `hardExpireAt` 后若仍有余额，会写入一条 `会员到期清零` 到 `courts.history`

### 2.4 当前系统中必须承认的实现约束

一期开发必须承认以下现实：

1. 当前没有独立 `benefit_batch` 表。
2. 当前补发权益实现方式，是写入 `membership_benefit_ledger`，并归属到一个已有购买批次。
3. 当前会员页面、订场用户页面、学员页面都在同一个 `public/index.html` 中实现。
4. 当前后端所有会员接口都在 `api/index.js` 中。

因此，一期不要同时做“大交互改造 + 大数据模型改造”。

## 3. 一期改造总目标

### 3.1 目标

一期目标只有 4 个：

1. 让管理员清楚知道“在哪里看、在哪里操作、这次操作会改什么”。
2. 把订场用户编辑和会员操作彻底隔离，消除误操作。
3. 让会员相关信息按频率和重要性分层展示，而不是继续堆表。
4. 在不大改底层模型的前提下，让前后端口径统一。

### 3.2 一期不追求

一期不追求：

1. 完美的数据模型重构
2. 把所有历史概念一次性清理干净
3. 引入新的会员批次表
4. 大规模拆分前后端文件结构

### 3.3 一期总策略

一期总策略：

1. 后端核心账务逻辑尽量不动
2. 页面结构和入口优先重构
3. 显示口径优先统一
4. 复杂模型升级放到二期

## 4. 统一业务定义

### 4.1 订场用户

订场用户是会员主体。  
所有会员开卡、续充、作废、折扣、余额、权益，最终都挂在订场用户上。

### 4.2 会员方案

会员方案是可售卖的标准模板，定义：

1. 方案名称
2. 充值金额
3. 折扣
4. 赠送金额
5. 赠送权益模板
6. 内部备注

会员方案不是成交记录。

### 4.3 会员账户

会员账户是某个订场用户当前正在生效的会员状态总览，回答：

1. 当前方案是什么
2. 当前折扣是什么
3. 当前余额是多少
4. 当前余额有效期到哪天
5. 最晚清零日到哪天
6. 当前是否还能进行续充、消耗权益、补发、作废

### 4.4 会员购买记录

会员购买记录是一次开卡或续充成交。  
它是成交事实，不是当前状态。

### 4.5 权益批次

一期定义下，权益批次是“由一条会员购买记录生成的一组权益快照”的展示概念。  
也就是说：

1. 一条购买记录对应一批基础赠送权益
2. 管理员需要能看到该批购买送了什么、还剩什么、何时到期
3. 该展示可以由 `membership_orders + membership_benefit_ledger` 推导，不要求一期新增独立批次表

### 4.6 权益流水

权益流水是权益变动记录，包括：

1. `grant`
2. `consume`
3. `supplement`
4. 后续可能扩展的 `void`、`expire`

权益流水是审计记录，不是首页主视图。

### 4.7 补发权益

一期正式定义：

1. 补发权益不生成新的购买记录
2. 补发权益不当作新的“会员购买批次”
3. 补发权益继续沿用当前模型，作为某个已有购买批次的权益调整记录
4. UI 必须诚实表达“这是补发调整，不是购买送的”

因此，一期不采用“补发新建独立批次”的设计。

原因：

1. 当前后端模型就是按已有购买批次挂补发流水实现的
2. 一期主要任务是整理交互，不是改账务底层
3. 如果补发强行改成独立批次，需要同时修改后端模型、汇总逻辑、扣减分配逻辑、历史数据兼容，爆炸半径过大

二期如业务确认有强审计需求，再单独升级。

## 5. 统一业务规则口径

### 5.1 余额规则

全系统统一按当前代码口径执行：

1. 每次会员购买会向 `courts.history` 写入一条 `会员充值`
2. 余额增加 = 充值金额 + 赠送金额
3. `validUntil = 购买日期 + 12个月`
4. `hardExpireAt = 购买日期 + 24个月`
5. 到 `validUntil` 后，如果余额仍大于 0，可进入延续期
6. 到 `hardExpireAt` 后，若余额仍大于 0，系统自动写入 `会员到期清零`
7. 会员页面只读展示余额，不允许直接编辑余额

### 5.2 折扣规则

1. 当前折扣按最近一次生效档位决定
2. 不叠加历史折扣
3. 同档续充，折扣不变
4. 升档续充，折扣更新为新档位
5. 低档续充，折扣更新为新档位
6. 折扣只影响后续订场扣款，不回改历史消费

### 5.3 有效期重置规则

1. 如果用户当前仍在有效期内，且本次充值金额不低于上一笔合规续充金额，则重置有效期
2. 如果本次充值金额低于上一笔合规续充金额，则允许充值，但不重置有效期
3. 这个结果必须在续充预览区明确展示，不能只在后端 silent warning

### 5.4 权益规则

1. 每次购买会生成一组权益快照
2. 每组权益有效期默认 12 个月
3. 权益消耗优先扣最早到期的批次
4. 原有权益不会因为续充或降档而消失
5. 低档续充不会抹掉高档时期已经送出的权益
6. 权益不能折现，不能并入余额

### 5.5 一次性调整规则

当前代码支持成交时做一次性权益调整，这个能力保留。

正式规则：

1. 一次性调整只影响本次购买记录
2. 不回写会员方案模板
3. 默认折叠，不作为弹窗首屏主内容
4. 调整区支持：
   - 实收金额
   - 赠送金额
   - 各项权益次数
   - 指定教练陪打次数
   - 指定教练范围
   - 自定义权益

### 5.6 作废规则

一期统一按当前后端能力定义：

1. 作废会员后，账户状态变为 `voided`
2. 余额不自动退款
3. 购买记录保留
4. 权益不可再继续使用
5. 后续如需重新启用，走“重新开卡”

### 5.7 重新开卡规则

一期重新开卡适用状态：

1. `voided`
2. `cleared`

不再把 `expired` 作为一期主状态口径。  
如果界面需要“已过期/即将到期”提示，应作为展示态派生，而不是强依赖数据库状态。

## 6. 需求边界：各模块与各端联动

## 6.1 订场用户模块

### 定位

订场用户模块是 CRM 与基础财务入口，不是会员主操作页。

### 必须保留

1. 姓名
2. 手机号
3. 校区
4. 关联学员
5. 对接人
6. 对储值态度
7. 熟悉程度
8. 备注
9. 当前余额
10. 累计充值 / 累计消费 / 累计实收
11. 充值消费记录

### 必须新增

1. 明确的 `查看账户` 入口
2. 行内或操作列上，将 `查看账户` 与 `设置` 分开

### 必须删除

1. 在“编辑订场用户”弹窗里直接执行会员操作的入口

### 边界

订场用户模块只负责：

1. 基本资料
2. CRM 跟进字段
3. 基础财务查看与录入
4. 跳转会员账户详情

不负责：

1. 开卡
2. 续充
3. 消耗权益
4. 补发权益
5. 作废会员

## 6.2 编辑订场用户弹窗

### 定位

纯资料编辑弹窗。

### 展示内容

1. 基本信息
2. 关联学员
3. 校区
4. 对接人
5. 对储值态度
6. 熟悉程度
7. 备注
8. 充值/消费记录只读区

### 明确不展示

1. 开通会员
2. 续充会员
3. 消耗权益
4. 补发权益
5. 作废会员
6. 最近购买记录
7. 权益操作按钮组

### 说明

如果需要展示会员状态，只允许展示只读摘要，不允许在此弹窗中发起会员操作。

## 6.3 会员管理首页

### 定位

会员管理首页是管理员的会员主操作页，默认视角必须是“会员账户”。

### 首页主目标

让管理员快速回答：

1. 谁是会员
2. 当前是什么档位
3. 谁快到期
4. 谁需要续充
5. 谁的权益有异常

### 顶部指标

一期固定展示：

1. 活跃会员数
2. 30 天内到期人数
3. 近 30 天续充金额
4. 权益异常数

### 首页筛选

一期保留最小必要集合：

1. 搜索：用户姓名 / 手机号 / 当前方案
2. 状态：全部 / 正常 / 延续期 / 已清零 / 已作废
3. 风险视角：30 天内到期

### 首页列表字段

一期固定为：

1. 订场用户
2. 当前方案
3. 会员状态
4. 当前余额
5. 当前折扣
6. 权益摘要
7. 余额有效期
8. 操作

### 首页辅助入口

以下内容不再作为首页主 tab，而作为辅助入口：

1. 方案配置
2. 全局购买记录
3. 全局权益流水

## 6.4 会员账户详情

### 定位

会员账户详情是单用户会员操作中心。  
所有会员动作都应从这里发起。

### 结构

账户详情固定分 4 个区块：

1. 当前状态
2. 权益批次
3. 最近购买记录
4. 操作按钮区

### 区块 1：当前状态

必须展示：

1. 当前方案名
2. 会员状态
3. 当前折扣
4. 当前余额
5. 余额有效期
6. 最晚清零日
7. 余额来源提示：来自 `courts.history`

### 区块 2：权益批次

必须按购买批次展示，不能只展示合计总数。

每个批次卡片至少显示：

1. 购买日期
2. 对应方案
3. 批次到期日
4. 各项权益剩余/原始赠送数
5. 如该项有补发，显示“含补发 +N”

示例表达：

1. 穿线免手工费：`7 / 原始5（含补发+2）`
2. 发球机免费：`4 / 原始6`

### 区块 3：最近购买记录

只显示最近 3 条，字段如下：

1. 购买日期
2. 方案名
3. 实收金额
4. 赠送金额
5. 本次是否重置有效期
6. 备注摘要

### 区块 4：操作按钮区

操作按钮按账户状态动态显示，不允许所有按钮同时出现。

## 6.5 方案配置模块

### 定位

方案配置是运营配置页，不是财务页，不是购买页。

### 运营可见字段

1. 方案名称

3. 折扣
4. 赠送金额
5. 大师公开课次数

8. 国家二级运动员陪打次数
9. 指定教练陪打次数
10. 指定教练范围
11. 自定义权益
12. 内部备注

### 系统隐藏字段

1. `tierCode`

3. `validMonths`
4. `maxMonths`
5. `createdAt`
6. `updatedAt`

### 方案页不负责

1. 实际开卡
2. 实际续充
3. 购买记录编辑

## 6.6 全局购买记录模块

### 定位

全局购买记录是审计页，不是主操作入口。

### 主要用途

1. 查某人什么时候买过什么
2. 查成交金额
3. 查一次性调整后的当次权益快照
4. 查是否重置有效期

### 不承担的职责

1. 从此页发起续充
2. 从此页直接补发或扣减权益

## 6.7 全局权益流水模块

### 定位

全局权益流水是追溯页，只负责解释“为什么变了”。

### 必须展示

1. 时间
2. 用户
3. 关联购买批次
4. 权益类型
5. 变动数量
6. 动作
7. 原因
8. 操作人

### 不承担的职责

1. 发起补发
2. 发起消耗
3. 发起作废

## 6.8 学员模块联动

### 定位

学员页面只展示“关联订场账户会员摘要”，不承担会员主操作。

### 允许展示

1. 关联订场用户姓名
2. 当前方案
3. 会员状态
4. 当前折扣
5. 到期信息

### 不允许展示

1. 开卡
2. 续充
3. 补发
4. 消耗
5. 购买记录编辑

## 6.9 教练端联动

### 规则

教练端继续不加载任何会员数据。

### 原因

1. 会员金额属于敏感运营数据
2. 会员购买与权益流水不属于教练操作范围
3. 当前后端与测试已经按这个权限模型设计

## 6.10 订场消费联动

### 一期边界

一期不新增新的订场结算流程，只沿用现有逻辑。

### 现有逻辑继续成立

1. 会员折扣影响订场消费扣款金额
2. 使用储值扣款时，按当前会员折扣计算实际扣款
3. 余额扣减仍发生在 `courts.history`
4. 会员页只展示结果，不直接做消费结算

## 7. 前端交互方案

## 7.1 信息层级

### L1：首页/列表层

用于快速判断是否要行动，只放：

1. 当前方案
2. 会员状态
3. 当前余额
4. 当前折扣
5. 余额有效期
6. 权益摘要

### L2：账户详情层

用于理解单个会员全貌，放：

1. 当前状态
2. 按批次权益
3. 最近购买记录
4. 操作按钮

### L3：操作弹窗层

用于执行动作，统一结构：

1. 标题
2. 当前状态摘要
3. 本次会发生什么
4. 折叠的一次性调整区
5. 参数输入
6. 确认按钮

### L4：审计层

用于查原因，不做高频操作入口：

1. 全局购买记录
2. 全局权益流水

## 7.2 首页与订场用户页的关系

1. 订场用户页负责“找人 + 资料 + 财务”
2. 会员管理页负责“会员行动与追踪”
3. 两边都可以进入账户详情
4. 但只有账户详情能发起会员操作

## 7.3 一次性调整区的交互

1. 默认折叠
2. 仅在管理员需要灵活成交时展开
3. 展开后必须用一行说明强调：
   - 只影响本次购买
   - 不修改会员方案

## 7.4 权益批次卡的交互

1. 默认按批次显示
2. 默认展示每个批次的核心权益状态
3. 可展开查看更多权益项目
4. 批次卡上不直接放“补发”按钮
5. 补发仍从账户详情统一操作按钮进入

## 7.5 消耗权益弹窗的交互

必须明确展示：

1. 当前所选权益类型
2. 当前总剩余
3. 将优先扣减哪一个批次
4. 如果当前批次不足，将继续扣下一批

管理员不需要手动选批次。

## 7.6 补发权益弹窗的交互

一期基于现有模型，补发弹窗必须明确写明：

1. 本次补发不会生成购买记录
2. 本次补发会归属到某个已有购买批次
3. 补发后，该批次对应权益的剩余数会增加
4. 补发原因必填

不要把补发伪装成“新批次购买”。

## 7.7 作废会员弹窗的交互

必须用危险态文案明确展示：

1. 折扣立即失效
2. 余额不会自动退款
3. 权益不可再继续使用
4. 购买记录和流水保留
5. 后续如需恢复，只能重新开卡

## 8. 场景化操作需求

## 8.1 首次开卡

### 触发条件

该订场用户从未有过会员账户记录。

### 入口

会员账户详情 → `首次开卡`

### 预览区必须展示

1. 方案名
2. 充值金额 + 赠送金额
3. 当前余额将增加多少
4. 折扣将变为什么
5. 余额有效期到哪天
6. 新增哪些权益
7. 当前无旧会员，无需处理旧权益

### 保存后影响

1. 新建会员账户
2. 新建购买记录
3. 写入 grant 权益流水
4. 写入 `courts.history` 的会员充值

## 8.2 正常续充

### 触发条件

当前账户状态为正常或延续期，且本次充值金额不低于上一笔合规续充金额。

### 预览区必须展示

1. 当前方案、当前余额、当前折扣、当前到期日
2. 本次余额增加多少
3. 有效期是否重置
4. 折扣是否变化
5. 原有权益全部保留
6. 本次新增哪些权益

## 8.3 低档续充

### 触发条件

当前账户有效，但本次充值金额低于上一笔合规续充金额，或所选方案低于当前档位。

### 规则

一期正式规则：

1. 允许低档续充
2. 余额照常增加
3. 折扣按新档位变更
4. 原有权益保留
5. 不重置有效期

### 预览区必须额外强调

1. 这是低档续充
2. 折扣如何变化
3. 本次不会重置有效期
4. 如需重置有效期，请改选不低于原档位的方案

## 8.4 重新开卡

### 触发条件

当前账户为：

1. `cleared`
2. `voided`

### 预览区必须展示

1. 上一张卡的状态摘要
2. 旧历史保留
3. 新卡按新的方案重新生效
4. 本次重新开卡后会新增哪些内容

## 8.5 消耗权益

### 触发条件

当前账户可用，且该权益还有剩余次数。

### 预览区必须展示

1. 所选权益
2. 当前总剩余
3. 当前将优先扣哪个批次
4. 扣减后剩余会变多少

### 保存后影响

1. 只写权益流水
2. 不影响余额
3. 不影响折扣
4. 不影响有效期

## 8.6 补发权益

### 触发条件

当前账户可用。

### 预览区必须展示

1. 补发的权益类型
2. 补发次数
3. 归属到哪个购买批次
4. 该批次该项权益会增加多少
5. 原因必填

### 保存后影响

1. 写 supplement 权益流水
2. 不写购买记录
3. 不改余额
4. 不改折扣
5. 不改有效期

## 8.7 作废会员

### 触发条件

当前账户为正常或延续期。

### 保存后影响

1. 账户状态更新为 `voided`
2. 会员折扣失效
3. 权益不再可用
4. 余额保留在 `courts.history`
5. 后续只能重新开卡

## 8.8 仅修改资料

### 规则

该场景始终与会员操作隔离。

### 结果

1. 只改订场用户资料
2. 会员相关任何表都不变化

## 9. 状态与按钮可见性

| 展示状态 | 判定方式 | 显示按钮 | 隐藏按钮 |
| --- | --- | --- | --- |
| 未开卡 | 无会员账户 | 首次开卡 | 续充、消耗、补发、作废 |
| 正常 | `active` | 续充会员、消耗权益、补发权益、查看流水、作废会员 | 首次开卡、重新开卡 |
| 即将到期 | `active/extended` 且 30 天内到期 | 续充会员（高亮）、消耗权益、补发权益、查看流水、作废会员 | 首次开卡、重新开卡 |
| 延续期 | `extended` | 续充会员、消耗权益、补发权益、查看流水、作废会员 | 首次开卡、重新开卡 |
| 已清零 | `cleared` | 重新开卡、查看流水 | 续充、消耗、补发、作废 |
| 已作废 | `voided` | 重新开卡、查看流水 | 续充、消耗、补发、作废 |

说明：

1. `即将到期` 是展示态，不要求单独落库存状态
2. 一期不以 `expired` 作为主状态口径

## 11. 开发实施要求

### 11.1 前端要求

1. 优先重构信息结构，不先堆字段
2. 所有会员写操作必须从账户详情发起
3. 预览区必须在前端即时计算并展示
4. 提交时仍由后端做最终校验

### 11.2 后端要求

一期后端原则：

1. 保持现有账务逻辑稳定
2. 不新增独立批次表
3. 明确返回低档续充 warning
4. 保持去重与校验逻辑

### 11.3 文案要求

全系统统一使用以下术语：

1. 订场用户
2. 会员方案
3. 会员账户
4. 会员购买记录
5. 权益批次
6. 权益流水
7. 余额有效期
8. 最晚清零日

禁止再混用：

1. 当前会员 = 订场会员
2. 会员类型 = 方案名
3. 权益批次 = 权益流水

## 12. 验收标准

满足以下条件，视为一期方案达标：

1. 管理员在“编辑订场用户”时，不会看到任何会员操作按钮
2. 管理员可以从订场用户列表或会员首页进入同一个会员账户详情
3. 会员首页默认看到的是会员账户总览，不是方案表
4. 管理员在续充前，可以清楚看到：
   - 余额怎么变
   - 折扣怎么变
   - 有效期是否重置
   - 原有权益是否保留
   - 新增权益是什么
5. 管理员在消耗权益时，可以看到将优先扣减哪一批
6. 管理员在补发权益时，可以明确知道这是“批次调整”，不是“新增购买”
7. 学员端只看到只读会员摘要
8. 教练端看不到任何会员数据
9. 一期不引入新的底层账务冲突

## 13. 结论

本次正式方案的核心不是“再做一个更漂亮的会员页”，而是：

1. 让会员操作从订场用户资料页中脱离出来
2. 让会员系统按管理员真实动作组织
3. 在不破坏现有后端账务逻辑的前提下，把信息讲清楚

后续开发必须优先遵守本文件中的：

1. 业务定义
2. 规则口径
3. 模块边界
4. 一期范围

如后续要做模型升级，必须在本文件基础上另开二期方案，不得在一期实现中临时扩张范围。
