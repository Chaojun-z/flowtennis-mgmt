# 线索池冻结方案与上线门禁

## 1. 文档目的

本文件用于冻结 `线索池` 模块的一期方案，明确：

- 什么能做
- 什么不能做
- 对现有功能、逻辑、数据的影响边界
- 线索池转化流转的固定规则
- 重复信息防控规则
- 开发、测试、导入、上线前必须通过的门禁

本文件是后续开发、测试、试导入、上线的统一依据。

---

## 2. 冻结原则

本次只允许做 3 件事：

1. 新增 `线索池`
2. 新增 `线索跟进时间线`
3. 新增 `线索 -> 学员 / 订场用户` 转化桥

本次明确不允许动：

- 课包购买主逻辑
- 班次 / 学习计划主逻辑
- 排课 / 扣课主逻辑
- 订场财务主逻辑
- 会员账户 / 权益主逻辑

结论：

`线索模块只加在现有系统前面，不插进现有核心业务中间。`

---

## 3. 最终结构冻结

一期只认下面这套结构，不再发散：

- `Lead`
  - 线索主档案
- `LeadFollowup`
  - 每次跟进记录
- `Student`
  - 课程身份，继续沿用现有学员体系
- `Court`
  - 订场身份，继续沿用现有订场体系
- `Membership`
  - 会员身份，继续挂在 `courtId` 上

关系冻结为：

- 一个 `Lead` 可以不转化
- 一个 `Lead` 可以转 `Student`
- 一个 `Lead` 可以转 `Court`
- 一个 `Lead` 可以同时转 `Student + Court`
- `Membership` 不从 `Lead` 直接创建，只通过 `Court` 识别

## 6. 重复数据防爆规则

### 6.1 唯一判断优先级

导入和转化都统一走下面这套规则：

1. 手机号完全一致 -> 一级主匹配
2. 姓名完全一致且手机号为空 -> 二级疑似匹配
3. 微信名 / 昵称相同 -> 只提示，不自动绑定

### 6.2 禁止事项

明确禁止：

- 仅凭微信名自动建学员
- 仅凭微信名自动绑订场用户
- 导入时自动把所有 `已报名` 直接写成学员
- 导入时自动把所有 `定场` 直接写成订场用户

### 6.3 匹配结果只能有三类

每条导入记录只能落到：

- `已自动关联`
- `疑似匹配待确认`
- `未匹配待处理`

禁止出现：

- `静默自动乱绑`

### 6.4 对象创建硬约束

- 一个 `Lead` 不能重复创建第二个 `Student`
- 一个 `Lead` 不能重复创建第二个 `Court`
- 已有关联时，只允许查看、改绑确认，不允许继续创建

## 7. 线索池流转冻结

### 7.1 线索录入

新线索进入后，只创建：

- `Lead`
- 首条 `LeadFollowup`

不创建：

- 学员
- 订场用户
- 会员账户

### 7.2 跟进

每次回访只新增 `LeadFollowup`，并自动回写 `Lead` 汇总字段：

- 最近跟进时间
- 最近结论
- 当前状态
- 下次跟进时间
- 下一步动作

### 7.3 转课程

只有点击：

- `转为学员`
- `关联已有学员`

时，才允许出现 `studentId`。

### 7.4 转订场

只有点击：

- `转为订场用户`
- `关联已有订场用户`

时，才允许出现 `courtId`。

### 7.5 转会员

线索层不做 `转会员` 按钮。

只做只读识别：

- 如果 `courtId` 已有会员账户，则 `Lead` 显示 `已升级会员`

这样可以避免线索层把会员逻辑打乱。

## 8. 状态冻结

### 8.1 原始状态

原样保留 CSV 的业务表达，例如：

- `已沟通`
- `体验课预约`
- `体验课完成`
- `已报名-私教`
- `已报名-训练营`
- `已定场`
- `无意向`
- `已流失`

### 8.2 系统状态

用于筛选和统计的系统状态只允许：

- `新线索`
- `跟进中`
- `已约体验`
- `已转课程`
- `已转订场`
- `已转课程+订场`
- `已流失`

归类规则冻结如下：

- 原始状态含 `已报名` -> `已转课程`
- 原始状态为 `已定场` -> `已转订场`
- 同时关联 `studentId + courtId` -> `已转课程+订场`
- 原始状态为 `已流失` / `无意向` -> `已流失`
- 原始状态为 `体验课预约` -> `已约体验`
- 其他默认 -> `跟进中`

## 9. CSV 导入冻结规则

### 9.1 预览导入

预览导入只允许做：

- 识别字段
- 校验缺列
- 生成状态统计
- 尝试匹配学员 / 订场 / 会员
- 生成预览报告

预览导入绝不写库。

### 9.2 正式导入

正式导入时每行只允许做：

1. 创建 `Lead`
2. 创建首条 `LeadFollowup`
3. 回填自动匹配到的 `studentId / courtId / membershipAccountId`
4. 标记导入结果

正式导入时明确不允许：

- 自动创建学员
- 自动创建订场用户
- 自动创建会员
- 自动修改现有学员数据
- 自动修改现有订场数据

## 10. 现有系统影响边界

### 10.1 学员体系

必须保持不变：

- 学员新增 / 编辑
- 学员详情
- 学员购买课包
- 学员班次 / 学习计划

只允许新增：

- 学员详情中的 `来源线索摘要` 只读展示

### 10.2 购买记录体系

必须保持不变：

- 新增购买记录
- 编辑购买记录
- 已有扣课记录时的锁定逻辑
- 学员详情里的课包摘要

线索模块不得直接写 `purchases`。

### 10.3 班次 / 学习计划 / 排课

必须保持不变：

- 班次创建和锁定规则
- 排课创建 / 编辑 / 取消
- 扣课推荐逻辑

线索模块不得直接写：

- `classes`
- `plans`
- `schedule`

### 10.4 订场体系

必须保持不变：

- 订场用户新增 / 编辑
- 财务流水
- 合并订场用户逻辑
- 关联学员能力

只允许新增：

- 订场详情中的 `来源线索摘要` 只读展示

### 10.5 会员体系

必须保持不变：

- 会员开户 / 续充
- 会员订单
- 权益流水
- 会员继续只挂 `courtId`

线索模块不得直接写：

- 会员订单
- 权益流水

## 11. 运营丝滑度验收标准

只有下面 8 条全部满足，才算转化流转足够丝滑：

1. 运营录入新线索不超过 1 分钟
2. 新增一次跟进不超过 30 秒
3. 不需要手写日期
4. 打开线索详情后 5 秒内能看懂最近情况
5. 能一眼看到 `下次什么时候跟`
6. 转为学员时不需要重复填一遍大部分信息
7. 转为订场用户时不需要重复填一遍大部分信息
8. 进入学员 / 订场详情后，能看到来源线索摘要

## 12. 上线门禁

### 门禁 A：结构门禁

必须确认：

- 只新增 `Lead + LeadFollowup`
- 不重写核心业务表
- 不改变会员挂载关系

### 门禁 B：数据门禁

必须确认：

- CSV 全字段可落
- 导入预览可跑
- 正式写库前不改任何现有对象
- 疑似匹配必须人工确认

### 门禁 C：流程门禁

必须确认下面路径全部走通：

- 新线索 -> 跟进 -> 转学员
- 新线索 -> 跟进 -> 转订场
- 新线索 -> 跟进 -> 流失
- 已有学员 / 已有订场关联路径

### 门禁 D：回归门禁

必须确认原有核心流程全绿：

- 学员
- 购买记录
- 班次
- 排课
- 订场
- 会员

### 门禁 E：试导入门禁

必须先用目标 CSV 跑完整试导入流程：

1. 预览
2. 人工抽查 20 条
3. 测试环境正式导入
4. 抽查学员 / 订场 / 会员关联结果
5. 确认无误后才允许生产导入

## 13. 最终冻结结论

后续开发只能按下面这句话执行：

`线索池是一个独立的前置运营台账，负责记录线索和跟进时间线，并通过受控转化桥连接现有学员体系与订场体系；它不改写课程、订场、会员核心业务逻辑。`

这句话是总约束，不允许偏离。
