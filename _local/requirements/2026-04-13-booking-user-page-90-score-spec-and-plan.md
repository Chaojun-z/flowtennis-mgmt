# 订场用户页面 90 分改造方案与执行计划

> 本文档只定义改造方案和执行计划，不直接修改页面代码。

## 0. 最新执行说明

基于当前仓库最新状态，本轮不再停留在方案阶段，而是直接按本文执行改造。

当前执行口径：

1. 先落地第一优先级问题：
   - 订场流水反查排课占场
   - 固定历史流水校区归属
   - 删除保护补会员引用
2. 同步落地低成本高收益体验项：
   - 手动重复客户提醒
   - 编辑资料与记一笔流水拆分
   - 轻量跟进字段
   - CSV 导入导出可靠性修正
3. 不改整体导航、不做大重构、不重做会员模块边界。

本文后续 Task 顺序即为实际实施顺序依据。

## 1. 页面定位

订场用户页不是普通学员页，也不是会员方案配置页。

它在平台里的定位是：

1. 找到一个订场客户。
2. 看清这个客户的基础资料、学员关联、余额、消费、会员状态。
3. 记录订场相关的收款、储值扣款、退款、冲正。
4. 从客户视角进入会员账户查看和操作。
5. 让排课表知道哪些场地时间已经被订场客户占用。

保留这个页面是必要的。真实网球俱乐部里，付钱订场的人、上课的学员、会员权益主体经常不是同一个人。如果把订场用户并入学员页，会把课程课包、场地消费、储值余额、会员权益混在一起。

## 2. 当前主要问题

以下问题按优先级排列。

### P0：订场新增没有反查排课占场

当前排课保存时会检查订场用户流水是否占场。

但在订场用户页新增“消费 / 订场”流水时，只检查了日期、时间、场地、余额，没有同步检查排课表中是否已经有课程占用同一个校区、同一个场地、同一时间。

风险：

1. 运营在订场用户页录入一条订场消费。
2. 同一时间该场地已经有课程排课。
3. 系统可能形成“双占场”。

这是本轮改造第一优先级。

### P0：编辑订场用户校区会影响历史订场冲突判断

订场流水里如果没有单独保存校区，后端会用订场用户当前校区兜底。

风险：

1. 一个客户原来在 A 校区有历史订场记录。
2. 后来资料里的客户校区被改成 B 校区。
3. 历史订场记录可能被当成 B 校区的占场记录。
4. 排课冲突判断会变脏。

### P1：手动新增没有重复客户提醒

CSV 导入有去重逻辑：手机号优先，否则按“姓名 + 校区”去重。

但手动新增订场用户时，只校验手机号格式，没有提醒可能重复。

风险：

1. 同一个客户被建成多个订场用户。
2. 余额、会员、学员关联分散。
3. 运营后续很难判断哪个账户才是真实账户。

### P1：删除保护没有显式检查会员引用

当前删除订场用户会检查财务流水和财务金额。

正常会员开卡会写入订场用户流水，所以大多数会员用户删不掉。但如果数据异常，比如会员账户存在而 `courts.history` 缺失，仍可能留下会员账户、购买记录、权益流水等孤立数据。

删除保护需要补上会员账户、会员购买记录、会员权益流水引用检查。

### P1：资料编辑和财务流水在同一个弹窗里，误操作风险偏高

当前“编辑资料”弹窗中同时包含：

1. 基础资料。
2. 学员关联。
3. 只读财务汇总。
4. 新增充值/消费/退款/冲正流水。

虽然会员开卡、续充、权益消耗已经拆到会员账户页，但资料和钱仍在同一个弹窗。运营只是想改对接人或备注时，也会看到加流水区域，误加财务流水的概率较高。

### P2：CSV 导入编码判断不稳

现在依次尝试 `utf-8 / gb18030 / gbk` 解码，但浏览器在某些乱码情况下不一定报错，可能导致 GBK 文件被当成 UTF-8 解析，列名识别失败。

风险：

1. 运营导入中文 CSV。
2. 预览字段错乱或大量字段为空。
3. 导入结果不可靠。

### P2：CSV 导出不是所有字段都做转义

当前导出中主要是备注字段做了引号处理，姓名、对接人、态度等字段如果包含逗号、换行，也可能造成 CSV 错列。

## 3. 90 分目标

90 分不是把页面做复杂，而是让运营日常使用更稳。

达成标准：

1. 订场记录不会和排课表形成双占场。
2. 修改客户资料不会污染历史订场记录的校区归属。
3. 新增客户时能提醒重复，但不强行阻断特殊情况。
4. 删除订场用户不会留下会员孤立数据。
5. 资料编辑和财务流水入口更清楚，降低误操作。
6. 导入导出对运营的 Excel/CSV 使用更可靠。
7. 页面仍然保持当前架构，不做大重构。

## 4. 信息层级设计

### 4.1 列表层级

列表保留“运营找人 + 快速判断状态”的定位。

建议列表主字段：

1. 姓名。
2. 手机号。
3. 校区。
4. 账户类型：普通 / 储值 / 会员 / 历史会员。
5. 当前会员。
6. 会员状态。
7. 当前折扣。
8. 会员到期。
9. 当前余额。
10. 最近消费日期。
11. 对接人。
12. 下次跟进日期。
13. 备注摘要。
14. 操作：查看账户、编辑资料、记一笔流水。

建议新增轻量字段：

1. `lastFollowUpDate`：最近跟进日期。
2. `nextFollowUpDate`：下次跟进日期。

不建议新增复杂标签系统、自动评分系统、复杂 CRM 流程。

### 4.2 查看账户层级

“查看账户”用于客户视角的只读总览 + 会员高频操作。

展示顺序：

1. 客户资料摘要：姓名、手机号、校区、关联学员、对接人。
2. 财务摘要：当前余额、累计充值、累计消费、累计实收。
3. 会员摘要：当前方案、会员状态、折扣、余额有效期、最晚清零日。
4. 权益批次：每批权益剩余、到期日。
5. 最近流水：最近 5 条充值/消费/退款/冲正。
6. 最近购买记录：最近 3 条会员购买。
7. 操作区：会员操作和查看流水。

### 4.3 编辑资料层级

“编辑资料”只负责资料，不负责新增财务流水。

字段：

1. 姓名。
2. 手机号。
3. 关联学员。
4. 校区。
5. 加入日期。
6. 对接人。
7. 对储值态度。
8. 熟悉程度。
9. 最近跟进日期。
10. 下次跟进日期。
11. 备注。

只读展示：

1. 当前余额。
2. 累计充值。
3. 累计消费。
4. 累计实收。

按钮：

1. 取消。
2. 保存资料。

删除按钮仍可保留，但必须走删除保护。

### 4.4 记一笔流水层级

新增单独入口“记一笔流水”。

字段：

1. 类型：充值、消费、退款、冲正。
2. 分类：订场、私教课、班课、训练营、退款、冲正、其他。
3. 支付方式：微信、支付宝、现金、转账、储值扣款、储值退款、其他。
4. 关联学员。
5. 日期。
6. 开始时间。
7. 结束时间。
8. 场地。
9. 金额。
10. 备注。

规则：

1. 只有“消费 + 订场”必须填写时间和场地。
2. “消费 + 订场”保存前必须检查排课表占场。
3. 订场流水必须把当时的 `campus` 写进流水里，不能依赖客户资料当前校区。
4. 储值扣款不能导致余额为负。
5. 退款不能导致累计实收为负。
6. 冲正不能超过已有消费。

## 5. 联动边界

### 5.1 订场用户页负责

1. 订场客户资料。
2. 学员关联。
3. 订场财务流水入口。
4. 会员账户入口。
5. 订场占场记录。
6. 跟进信息。

### 5.2 会员管理页负责

1. 会员方案配置。
2. 会员购买记录审计。
3. 会员账户汇总。
4. 赠送权益流水审计。

订场用户页可以进入会员账户操作，但不要把会员方案配置塞进订场用户页。

### 5.3 学员页负责

1. 学员资料。
2. 课程班次、学习计划、课包权益。
3. 只读展示关联订场账户和会员摘要。

学员页不直接改订场用户余额，不直接开卡。

### 5.4 排课表负责

1. 课程排课。
2. 教练、学员、场地冲突检查。
3. 读取订场用户的订场流水作为场地占用来源。

排课表不编辑订场用户。

### 5.5 校区管理负责

1. 校区增删改。
2. 如果校区被订场用户、排课、学员、教练等引用，不能删除。

订场用户页只选择校区，不管理校区。

## 6. 执行计划

### Task 1：补后端订场流水与排课冲突校验

目标：从根上防止订场用户页新增订场记录和排课表双占场。

修改文件：

1. `api/index.js`
2. `tests/schedule-rules.test.js`
3. `tests/court-finance.test.js`

执行步骤：

1. 在后端增加一个可复用校验函数，例如 `validateCourtHistoryBookingConflicts(nextCourt, scheduleRows)`。
2. 函数只检查 `type === '消费'` 且 `category === '订场'` 的流水。
3. 对每条订场流水生成场地占用时间段。
4. 和排课表中有效排课做同校区、同场地、时间重叠校验。
5. 在 `/courts` 新增和 `/courts/:id` 更新保存前调用该函数。
6. 测试用例覆盖：排课已有 09:00-10:00，订场流水新增 09:30-10:30，应失败。
7. 测试用例覆盖：不同校区、不同场地、不重叠时间，应通过。

预期结果：

1. 排课保存会检查订场用户占场。
2. 订场用户新增订场流水也会检查排课占场。
3. 双向一致。

### Task 2：固定订场流水的校区归属

目标：历史订场记录不受客户资料校区变更影响。

修改文件：

1. `public/index.html`
2. `api/index.js`
3. `tests/court-finance.test.js`
4. `tests/schedule-rules.test.js`

执行步骤：

1. 前端新增流水时，继续把当前表单校区写入流水 `campus`。
2. 后端 `normalizeCourtHistory` 保留流水自己的 `campus`。
3. 订场冲突判断优先使用流水 `campus`。
4. 如果旧流水没有 `campus`，才使用客户当前 `campus` 兜底。
5. 新增测试：客户当前校区被改成 B，但历史流水已有 `campus: A`，排课冲突仍按 A 判断。

预期结果：

历史订场记录有自己的校区，不会被客户资料变更污染。

### Task 3：新增手动重复客户提醒

目标：减少重复订场用户。

修改文件：

1. `public/index.html`
2. `api/index.js`
3. `tests/court-page-view.test.js`：新增订场用户页面视图测试；如果实施时已有同类文件，则扩展该文件。

执行步骤：

1. 前端保存前按手机号查重。
2. 如果手机号为空，则按“姓名 + 校区”查重。
3. 编辑当前用户时排除自己。
4. 发现疑似重复时弹确认：
   - `系统中可能已有同一订场用户：xxx。是否仍然保存？`
5. 用户确认后允许继续保存。
6. 后端增加同样的软校验返回能力，或至少在后端保留可复用查重函数，避免未来其他入口绕过。

预期结果：

运营能被提醒，但特殊情况仍能继续保存。

### Task 4：补删除保护的会员引用检查

目标：避免删除订场用户后留下会员孤立数据。

修改文件：

1. `api/index.js`
2. `tests/membership-rules.test.js`

执行步骤：

1. 扩展 `assertCanDeleteCourt`，增加可选引用数据参数。
2. 删除单个订场用户时扫描：
   - `ft_membership_accounts`
   - `ft_membership_orders`
   - `ft_membership_benefit_ledger`
   - `ft_membership_account_events`
3. 如果存在 `courtId` 引用，返回错误：`该订场用户已有会员账户或会员流水，不能直接删除`。
4. 批量删除走同一套保护逻辑。
5. 测试覆盖：无财务流水但有会员账户时不能删除。

预期结果：

订场用户删除不会破坏会员链路。

### Task 5：拆分“编辑资料”和“记一笔流水”

目标：降低运营误操作。

修改文件：

1. `public/index.html`
2. `tests/membership-view.test.js`
3. `tests/court-page-view.test.js`：新增或扩展订场用户页面视图测试。

执行步骤：

1. 列表操作区改为：
   - 查看账户
   - 编辑资料
   - 记一笔流水
2. `openCourtModal` 保留资料编辑，不再放新增流水输入区。
3. 新增 `openCourtLedgerEntryModal(courtId)`，只处理新增充值/消费/退款/冲正流水。
4. 保存资料时只提交资料字段和原有历史流水，不新增流水。
5. 保存流水时只追加一条新流水，并重新计算财务金额。
6. 保留原有财务确认文案。
7. 视图测试检查“编辑资料”弹窗不再包含“添加流水”按钮或金额输入。

预期结果：

运营改资料和记账分开，页面更清楚。

### Task 6：新增轻量跟进字段

目标：让运营知道谁该跟进，不做复杂 CRM。

修改文件：

1. `public/index.html`
2. `api/index.js`
3. `tests/court-import-export.test.js`：新增导入导出解析和转义测试。

字段：

1. `lastFollowUpDate`
2. `nextFollowUpDate`

执行步骤：

1. 编辑资料弹窗增加最近跟进日期、下次跟进日期。
2. 列表增加下次跟进日期列。
3. 搜索覆盖这两个字段。
4. CSV 导入支持列名：最近跟进日期、下次跟进日期。
5. CSV 导出包含这两个字段。
6. 后端保存时保留字段，不做复杂自动计算。

预期结果：

运营能直接筛查和维护客户跟进节奏。

### Task 7：修 CSV 导入/导出可靠性

目标：减少运营导入导出错列。

修改文件：

1. `public/index.html`
2. `tests/court-import-export.test.js`

执行步骤：

1. 封装 `csvCell(value)`，所有导出字段统一转义。
2. 字段含逗号、引号、换行时，统一用双引号包裹并转义内部双引号。
3. 导入时优化编码判断，优先使用 `TextDecoder('utf-8', { fatal: true })`，失败后再尝试 `gb18030`。
4. 增加测试：姓名或备注包含逗号、换行、引号时导出不乱列。
5. 增加测试：中文 GBK/GB18030 文件能正确识别表头。

预期结果：

CSV 对 Excel 用户更稳定。

### Task 8：回归联动测试

目标：确保改造没有破坏其他页面。

执行命令：

```bash
node tests/court-finance.test.js
node tests/schedule-rules.test.js
node tests/membership-view.test.js
node tests/membership-rules.test.js
node tests/student-page-view.test.js
node tests/student-rules.test.js
```

重点检查：

1. 订场用户财务金额仍以流水为准。
2. 会员开卡/续充仍写入 `courts.history`。
3. 学员页仍能展示关联订场账户摘要。
4. 排课和订场占场双向互斥。
5. 删除保护不会误删会员链路。

## 7. 建议实施顺序

第一批必须先做：

1. Task 1：订场流水反查排课冲突。
2. Task 2：固定流水校区归属。
3. Task 4：删除保护补会员引用。

第二批做体验和运营效率：

1. Task 3：重复客户提醒。
2. Task 5：拆分编辑资料和记一笔流水。
3. Task 6：轻量跟进字段。

第三批做导入导出稳定性：

1. Task 7：CSV 导入导出可靠性。
2. Task 8：完整回归测试。

## 8. 不做的事

本轮不要做：

1. 不做复杂 CRM 自动评分。
2. 不做复杂客户标签系统。
3. 不做大范围页面重构。
4. 不把会员方案配置塞进订场用户页。
5. 不把订场用户并入学员页。
6. 不改平台整体导航和权限体系。

## 9. 验收标准

改造完成后，按下面标准验收：

1. 在排课表已有占场时，订场用户页新增同时间同场地订场流水会失败。
2. 在订场用户页已有订场流水时，排课表新增同时间同场地排课会失败。
3. 修改订场用户校区，不会改变已有流水的校区归属。
4. 新增疑似重复客户时有提醒。
5. 有会员账户、会员购买记录或会员权益流水的订场用户不能删除。
6. 编辑资料弹窗不再承担新增财务流水。
7. 记一笔流水入口能完成原有充值、消费、退款、冲正逻辑。
