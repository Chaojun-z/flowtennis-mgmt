# 管理端左侧菜单整改实施计划

> 目标：在不破坏现有业务功能的前提下，完成管理端左侧菜单重组、入口收口、页面职责梳理，以及“购买记录并入收入流水”的整改。

## 一、整改目标

本次整改只解决 4 件事：

1. 左侧菜单重新分组，消除“同类内容散落在不同模块”的问题。
2. 页面职责重新划线，避免 `订场用户 / 会员管理 / 财务中心` 之间继续互相抢职责。
3. 把各类“全局购买记录”收口到 `收入流水`，不再继续扩张左侧菜单。
4. 保留各业务页里的“当前对象局部记录”，避免为了收口全局入口把高频操作做坏。

本次整改不做：

1. 不改核心业务数据模型。
2. 不重做会员、课包、订场、财务的底层计算逻辑。
3. 不为了菜单整改做大范围页面重构。
4. 不新增复杂权限系统。

### 2. 页面职责边界

负责：

- 管理订场账户资料
- 看当前储值/消费摘要
- 看当前用户相关的订场记录

#### 会员管理

负责：

- 会员账户列表

- 当前会员的最近购买记录
- 当前会员的权益流水

不负责：

- 全局会员购买记录审计
- 全局权益流水审计

#### 售卖课包

负责：

- 课包售卖主操作

不负责：

#### 排课表

负责：

- 新增排课
- 编辑排课
- 取消排课
- 以课程、班次、学员为主线安排实际上课

不负责：

- 教练工作量统计
- 教练资源分析

#### 教练运营

负责：

- 教练排班视图
- 教练工作量统计
- 从教练维度查看排课分布、冲突、忙闲情况

不负责：

- 替代 `排课表` 成为唯一主排课入口
- 承担教学中心的课程执行主流程

#### 财务中心

负责：

- 全局收入审计
- 全局已入账视图
- 全局账务汇总
- 教练结算

## 三、核心整改原则

后续所有页面都按下面 3 条执行：

1. `左侧菜单` 只按业务域分，不按表格种类分。
2. `业务页面` 只保留当前对象相关记录，不再承担全局审计职责。
3. `全局查账` 全部收口到财务中心，不再新增“会员购买记录 / 课包购买记录 / 权益总流水”这类左侧独立菜单。

## 五、实施分期

### Phase 0：先定排课与教练运营边界

目标：先把 `班次管理 / 排课表 / 教练运营` 的职责关系定清楚，避免后面菜单整改时出现双主入口。

最终建议：

1. `教学中心 > 排课表`
   - 这是唯一标准排课主入口
   - “添加排课”统一从这里理解和落地
2. `资源管理 > 教练运营`
   - 页面内保留 2 个视图：
     - 教练排班视图
     - 教练工作量统计
   - 这是教练资源视角，不是教学主操作页
3. `教学中心 > 班次管理`
   - 管班次规则、班级容器和周期基础
   - 不直接替代实际排课页面

为什么这样放：

1. `班次管理` 管的是“班怎么建”。
2. `排课表` 管的是“课怎么上”。
3. `教练运营` 管的是“教练怎么排、工作量怎样”。

这 3 个页面是上下游关系，不是平级重复关系。

动作：

1. 保留 `schedule` 页面作为主排课页。
2. 保留 `coachops` 页面，但明确它是教练视角页。
3. 保留 `coachops` 内的“快捷新增排课”，但底层继续调用 `schedule` 同一套排课能力，不再发展成第二套主流程。
4. 在菜单、标题、按钮文案上明确：
   - `排课表` = 主排课
   - `教练运营` = 教练视图 + 工作量

验收标准：

1. 运营人员知道“添加排课”应去 `排课表`。
2. 教练运营里的排班视图不会被误解为独立排课系统。
3. 同一条排课数据仍然只走一套保存逻辑。

风险控制：

1. 先定入口，不先重构页面。
2. 不拆第二套排课数据结构。
3. 不删除教练运营现有快捷能力，只调整定位。

### Phase 1：只改导航，不改业务逻辑

目标：先把左侧看起来理顺，但页面功能先不搬，避免一次性动太多。

动作：

1. 调整左侧分组标题和排序：
   - 用户中心
   - 教学中心
   - 场地运营
   - 资源管理
   - 财务中心
2. 菜单文案统一：
   - `学员信息` -> `学员管理`
   - `排课表` 保持独立入口，放在 `教学中心`
   - `价格管理` -> `价格方案`
   - `会员方案` 如已存在，统一改成 `订场会员方案`
3. 左侧移除独立入口：
   - `购买记录`
   - `会员购买记录`
   - `会员权益总流水`
4. 财务中心内部保留现有 tab 或子视图，先挂载旧能力，做到“入口变了，能力没丢”。

验收标准：

1. 左侧结构和定稿一致。
2. 所有原有页面还能通过跳转进入。
3. 没有因为菜单删除导致找不到旧功能。

风险控制：

1. 这一期不删页面，只删左侧入口。
2. 旧页面保留路由/`goPage` 能力，避免硬切导致断链。

---

### Phase 2：重划页面职责，做“局部记录”和“全局记录”分层

目标：把业务页和财务页的边界切清楚。

动作：

1. `排课表` 定位为排课主页面：
   - 统一承接新增、编辑、取消排课
   - 保持来自班次、学员、学习计划、教练运营的跳转能力
2. `教练运营` 只保留两个内部视图：
   - 教练排班视图
   - 教练工作量统计
3. `教练排班视图` 只做教练资源观察和快捷操作：
   - 可查看教练维度的日/周/月排班
   - 可快捷发起新增排课
   - 但不再承担“全站排课主页面”职责
4. `会员管理` 页面只保留：
   - 当前会员最近购买记录
   - 当前会员权益流水
   - 跳转“查看全部收入流水”
   - 跳转“查看总账相关记录”
5. `售卖课包` 页面只保留：
   - 当前学员或当前课包相关购买记录
   - 余额/课时相关摘要
6. `订场用户` 页面只保留：
   - 当前用户相关订场记录
   - 当前用户储值/扣款摘要
7. 所有“查看全部”类入口统一跳财务中心，不再跳各自独立全局记录页。

验收标准：

1. 用户在业务页里还能看到和当前对象相关的记录。
2. 用户查全局记录时只需要去财务中心。
3. 业务页没有承担全局筛选和全局审计职责。
4. “添加排课”在用户认知里只有一个主入口：`教学中心 > 排课表`。
5. `教练运营` 仍能承担教练维度排班和工作量观察，但不与主排课流程抢职责。

风险控制：

1. 不先删局部记录，只先删“全局审计型按钮”。
2. 业务页保留熟悉的上下文记录，减少运营学习成本。

---

### Phase 3：收入流水收口

目标：把“购买记录”真正并入 `收入流水`。

动作：

1. 给 `收入流水` 增加统一筛选：
   - 业务类型
   - 用户
   - 校区
   - 时间
   - 支付方式
   - 状态
   - 操作人
2. 业务类型至少支持：
   - 课包购买
   - 会员购买
   - 订场收入
   - 渠道核销
3. 原 `purchases` 页面里的核心字段和视图能力并入 `收入流水`。
4. 原 `membership-orders` 页面里的全局会员购买记录能力并入 `收入流水` 的业务类型筛选。

验收标准：

1. 原来在“购买记录”里能查到的内容，在 `收入流水` 里都能查到。
2. 原来在“会员购买记录”里能查到的全局成交记录，在 `收入流水` 的会员筛选里能查到。
3. 不出现“并入后字段丢失”的情况。

风险控制：

1. 合并前先列字段对照表，不允许盲合。
2. 合并初期允许保留旧页面作为只读兼容页。

---

### Phase 4：权益流水与总账关系收口

目标：把“变动类记录”放到合适位置，但不急着扩左侧菜单。

动作：

1. `会员管理` 继续保留“当前会员权益流水”。
2. 全局 `membership-ledger` 不再作为左侧入口。
3. 如果当前 `总账` 已有 tab 结构，则新增或复用内部标签承接“变动类记录”：
   - 权益变动
   - 余额变动
   - 其他后续变动
4. 如果本轮不适合并进总账，就先保留旧页能力，但只通过“查看全部相关记录”跳转进入，不挂侧栏。

验收标准：

1. 当前会员的权益流水仍可查看。
2. 全局权益流水不再污染侧栏。
3. 变动类查询有明确落点。

风险控制：

1. 这一步允许延后，不强行一次做完。
2. 如果总账当前能力不足，先做“入口收口”，后做真正数据融合。

## 八、推荐执行顺序

按下面顺序做，最稳：

1. 改左侧菜单分组和文案
2. 先定 `班次管理 / 排课表 / 教练运营` 三者入口关系
3. 移除侧栏冗余入口，但保留旧页面
4. 调整业务页“查看全部”跳转
5. 给财务中心补收入流水筛选能力
6. 把 `purchases` 和 `membership-orders` 的全局能力并入收入流水
7. 最后再处理 `membership-ledger` 和 `总账` 的关系

原因：

1. 先改导航，用户先感知到“变清晰了”。
2. 先把排课主入口定死，避免教学中心和资源管理继续抢同一件事。
3. 先入口收口，再做能力融合，风险更低。
4. 把最复杂的“变动类流水”放最后，避免前面改炸。

---

## 九、最终落地口径

最终落地以后，整个系统按下面口径解释：

1. `左侧菜单` 只负责告诉用户“去哪个业务域办事”。
2. `业务页` 只负责当前对象的操作和上下文记录。
3. `财务中心` 只负责全局查账、全局收入、结算与审计。
4. `购买记录` 不再是独立业务域，而是 `收入流水` 的一种类型。
5. `权益流水` 继续存在，但属于当前会员上下文或总账内部能力，不再单独占左侧入口。

## 十、建议的执行方式

建议拆成 2 个开发批次：

### 批次 A：低风险信息架构整改

- 菜单分组
- 菜单文案
- 排课表 / 教练运营 / 班次管理归位
- 入口迁移
- 跳转调整

### 批次 B：中风险流水收口整改

- 收入流水合并
- 旧全局记录页降级
- 总账与权益流水关系梳理

这样做的好处是：

1. 第一批先把“乱”解决。
2. 第二批再解决“散”。
3. 就算第二批延期，第一批也已经能明显改善使用体验。
