# FlowTennis 约球小程序与后台联动正式规划

## 1. 文档目的

本文档用于明确 FlowTennis 约球小程序一期正式版的产品范围、后台联动方式、财务接入方式、数据结构、状态流转、开发排期与验收标准。

后续开发、测试、产品确认、财务联调，均以本文档为主。

本文档对应方案为：

- 小程序负责约球闭环
- 现有后台新增运营订场工作台
- 财务模块接收自动生成的待确认应收记录
- 财务人工确认入账
- 一期不做微信支付、不做自动订场、不做自动分账

---

## 2. 当前业务现状

### 2.1 现状流程

当前业务实际流程如下：

1. 用户有约球需求
2. 运营或工作人员先协调可行时间段
3. 用户报名参与
4. 运营去第三方订场小程序下单订场
5. 第三方系统不返回订场数据给 FlowTennis
6. 运营再回到现有管理后台手工录入订场和费用信息
7. 财务再根据人工录入数据进行后续处理

### 2.2 当前核心问题

- 约球数据、报名数据、订场数据、财务数据分散
- 第三方订场系统无法打通
- 运营重复录入，效率低
- 财务数据来源链路不清晰
- 用户参与记录无法完整沉淀到自有后台
- 后续自建订场能力缺少统一底座

### 2.3 一期建设目标

一期不是直接替代第三方订场系统，而是先把这条链路收回到自己系统里：

`发起约球 -> 用户报名 -> 运营确认并订场 -> 系统生成AA费用 -> 财务待确认 -> 人工入账`

---

## 3. 一期方案结论

本项目采用以下推荐方案：

### 3.1 小程序侧

小程序负责完成用户侧约球闭环：

- 微信登录
- 浏览约球
- 发起约球
- 报名约球
- 取消报名
- 查看我的约球
- 查看约球详情
- 接收基础通知

### 3.2 后台侧

基于现有 `flowtennis-mgmt-main` 继续扩展，不新建独立后台。

后台新增：

- 约球管理模块
- 运营订场工作台
- AA结算中心
- 财务待确认模块

### 3.3 财务侧

系统自动生成财务待确认记录，但不自动入正式账。

财务人员在后台确认后才算入账。

### 3.4 一期明确不做

- 用户侧直接订场
- 对接第三方订场系统
- 微信支付
- 自动退款
- 自动分账
- 无人值守自动记账

## 4. 项目目标与边界

### 4.1 项目目标

一期目标分四层：

1. 用户侧建立约球闭环
2. 运营侧建立订场登记闭环
3. 财务侧建立应收确认闭环
4. 数据侧建立用户参与沉淀闭环

### 4.2 成功标准

上线后至少满足：

- 用户能在小程序里独立发起约球和报名
- 运营能在后台看到完整报名情况
- 运营能基于约球单录入订场结果
- 系统能按规则自动计算每人应收
- 系统能自动把应收推送到财务待确认列表
- 财务能按约球来源确认入账
- 用户参与记录能沉淀到后台和用户画像

### 4.3 项目边界

本期只覆盖“约球业务联动”，不覆盖“真实场地库存管理”和“真实支付清算”。

## 5. 用户角色定义

### 5.1 C端用户

使用小程序发起和参与约球的普通用户。

### 5.2 运营人员

在后台管理约球、协调人数、登记订场结果、录入费用的工作人员。

### 5.3 财务人员

在后台确认待入账记录、核对金额、导出对账数据的工作人员。

### 5.4 管理员

配置规则、查看全局数据、处理异常记录的内部管理角色。

---

## 6. 核心业务流程

### 6.1 主流程

1. 用户发起约球
2. 系统生成约球单，状态为 `open`
3. 其他用户报名
4. 人数满足条件后，运营在后台处理该约球单
5. 运营确认时间、人员，并线下完成订场
6. 运营在后台录入订场结果和场地费用
7. 系统根据 AA 规则拆分每位参与人的应收金额
8. 系统自动生成财务待确认记录
9. 财务在后台确认入账
10. 约球完成后沉淀到用户历史和统计报表

### 6.2 异常流程

需要支持以下异常场景：

- 用户报名后取消
- 运营取消整场约球
- 场地未订成功
- 实际到场人数与报名人数不一致
- 某位用户减免或免单
- 某位用户需补收差额
- 财务记录生成后被作废

## 7. 业务状态设计

### 7.1 约球单状态

建议使用以下状态：

| 状态 | 含义 | 说明 |
| --- | --- | --- |
| `draft` | 草稿 | 用户未正式发布 |
| `open` | 招募中 | 已发布，可报名 |
| `full` | 已满员 | 已达到目标人数 |
| `booking_pending` | 待订场 | 运营待处理 |
| `booked` | 已订场 | 已录入订场结果 |
| `completed` | 已完成 | 本次活动已结束 |
| `cancelled` | 已取消 | 整场取消 |

推荐的一期流转：

`draft -> open -> full -> booking_pending -> booked -> completed`

异常分支：

- 任意阶段可进入 `cancelled`

### 7.2 报名记录状态

| 状态 | 含义 |
| --- | --- |
| `registered` | 已报名 |
| `waitlisted` | 候补 |
| `cancelled` | 用户主动取消 |
| `removed` | 运营移除 |
| `attended` | 实际参加 |
| `absent` | 未到场 |

### 7.3 订场记录状态

| 状态 | 含义 |
| --- | --- |
| `pending` | 待录入 |
| `submitted` | 已提交订场信息 |
| `confirmed` | 已确认订场完成 |
| `failed` | 订场失败 |
| `cancelled` | 订场取消 |

### 7.4 财务记录状态

| 状态 | 含义 |
| --- | --- |
| `pending` | 待财务确认 |
| `confirmed` | 已确认入账 |
| `waived` | 已减免 |
| `void` | 已作废 |

## 10. 财务联动规则

### 10.1 一期财务原则

一期采用：

- 系统自动生成应收
- 财务人工确认入账

不采用：

- 自动支付
- 自动回调入账
- 自动退款

### 10.2 应收生成时机

建议一期使用以下规则：

1. 运营录入订场成功和总费用后
2. 运营确认实际参加名单
3. 系统生成费用拆分结果
4. 系统自动生成财务待确认记录

即：

`不是报名即生成应收`

而是：

`订场完成 + 实际参与确认后生成应收`

这样更贴近真实财务口径，也更适合避免后续反复冲销。

### 10.3 一期拆分规则

默认采用：

- 实际参加人数平均分摊

允许例外：

- 某用户减免
- 某用户免单
- 某用户补差额

所有例外必须记录：

- 调整前金额
- 调整后金额
- 调整人
- 调整原因

### 10.4 财务记录来源字段

财务记录至少需要保留：

- 来源类型：`match_fee_split`
- 来源约球单 ID
- 来源费用记录 ID
- 来源用户 ID
- 应收金额
- 调整金额
- 最终金额
- 当前状态

## 12. 接口规划

### 12.1 技术基础

当前系统技术现状：

- 前端：单页 HTML
- 后端：Vercel Serverless
- 数据库：阿里云 TableStore
- 小程序：当前仅有套壳项目

一期建议继续沿用现有架构，不新开第二套服务。

### 12.2 小程序端核心接口

建议新增以下接口能力：

- 用户登录/绑定
- 获取约球列表
- 获取约球详情
- 发起约球
- 报名约球
- 取消报名
- 获取我的约球
- 获取通知列表

### 12.3 后台端核心接口

建议新增以下接口能力：

- 约球单查询
- 约球单详情
- 运营录入订场结果
- 更新约球状态
- 确认实际参与名单
- 创建费用主记录
- 生成费用拆分
- 查询财务待确认记录

## 15. 开发量评估

以下评估基于：

- 继续使用现有后台
- 不做微信支付
- 不做自动订场
- 不接第三方订场系统

### 15.1 小程序端

- 页面与视觉还原：2 至 3 周
- 用户流程与接口联调：1 至 2 周

### 15.2 后台端

- 约球管理：1 至 1.5 周
- 运营订场工作台：1.5 至 2 周
- AA结算与财务待确认：1.5 至 2.5 周

### 15.3 后端与数据层

- 表结构与接口：2 至 3 周
- 权限与日志：0.5 至 1 周

### 15.4 测试与修正

- 流程联调：1 周
- 异常修正：0.5 至 1 周

### 15.5 总体工期

单人主力开发预计：

- 6 至 10 周

有前后端配合预计：

- 4 至 7 周

## 17. 风险点

### 17.1 业务规则反复变动

如果以下规则迟迟不定，会直接拖慢开发：

- 报名截止时间
- 满员后是否允许候补
- 取消后是否收费
- 费用按报名人数还是实际到场人数分摊
- 运营何时确认实际参加名单

### 17.2 财务口径不一致

必须提前定：

- 何时生成应收
- 作废如何处理
- 减免谁审批

### 17.3 后续突然加入支付

一旦一期中途加入微信支付、退款、分账，项目复杂度会明显增加，排期也会整体后移。

## 18. 一期验收标准

### 18.1 小程序验收

- 用户可登录
- 用户可发起约球
- 用户可报名和取消报名
- 用户可查看我的约球
- 用户可查看约球详情

### 18.2 后台验收

- 运营可查看约球单
- 运营可录入订场结果
- 运营可录入场地总费用
- 系统可生成 AA 拆分结果
- 财务可查看待确认应收
- 财务可确认入账

### 18.3 数据验收

- 每条应收能追溯到约球单
- 每条费用拆分能追溯到原始费用主记录
- 每条关键操作有日志
- 用户画像能累计参与次数和费用数据

---

## 19. 一期结论

本项目一期最合适的落地方式不是直接做“用户自己订场”，而是先围绕现有运营流程，完成以下闭环：

- 用户在小程序完成约球协同
- 运营在后台完成订场录入
- 系统自动完成 AA 费用拆分
- 财务在后台确认入账

这个方案最符合 FlowTennis 当前实际流程，改造成本可控，也为未来自建订场系统预留了统一的数据底座。
