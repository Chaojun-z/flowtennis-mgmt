# 学习计划页面 90 分改造方案与执行计划

## 1. 文档目的

本文档用于指导「学习计划页面」从当前的“班次进度台账页”，升级为更符合真实网球俱乐部管理员、教务、课程顾问、运营人员日常使用的“学习跟进页”。

本次改造坚持 4 个原则：

1. 先修正当前逻辑问题和数据口径风险，再做体验升级。
2. 信息层级要清楚，列表负责判断，详情负责理解，编辑归属原业务页处理。
3. 严格控制联动边界，学习计划页不变成班次页、排课页、权益页的混合体。
4. 不做复杂预警中心，不做花哨功能，只做真实门店能每天用上的内容。

---

## 2. 当前页面定位

当前页面已经具备基础能力：

- 展示学习计划列表
- 搜索学员 / 班次 / 关联账户
- 按状态筛选
- 展示课时进度
- 展示教练、课程、状态
- 由班次自动生成学习计划
- 随排课联动更新课时

当前真实定位更接近：

`学习计划台账页`

而不是：

`学习跟进管理页`

也就是说，它能“列出学员在班次里的记录”，但还不够支撑“跟进学员、安排下一步动作、辅助续费和排课决策”。

### P0-1. 课时进度条逻辑方向错误

当前页面把进度条宽度按“剩余课时 / 总课时”计算，而不是按“已上课时 / 总课时”计算。

风险：

- 页面标题写的是“课时进度”，但视觉表达实际是“剩余比例”。
- 管理员会误以为条越长代表上得越多，造成误判。
- 续费、结课、排课判断会被误导。

目标：

- 进度条统一表达“已上 / 总课时”。
- 同时保留右侧“剩余 X 节”的文字，不再让视觉和文字相互打架。

### P0-2. 学习计划展示的课时余额，不是当前最真实余额来源

当前排课已经同步扣减权益账户，但学习计划页展示的仍主要是旧的班次 / 学习计划课时。

风险：

- 页面显示“还有课”，但真实课包可能已到期、已用完、或不匹配当前教练 / 校区。
- 运营根据学习计划页判断是否还能继续排课，容易做错决策。
- 学习计划页和权益账户页口径分裂，降低系统可信度。

目标：

- 第一版先保留班次进度视角。
- 同时补充“真实权益摘要”，让管理员至少能看到当前可用权益与到期信息。
- 明确学习计划页不是最终余额结算页。

### P0-3. 班次允许手工编辑已上课时，导致学习计划进度可能失真

当前班次弹窗允许直接填写 `usedLessons`，而排课保存 / 编辑 / 删除也会自动加减课时。

风险：

- 班次进度和真实排课记录可能不一致。
- 学习计划同步出来的课时会跟着失真。
- 教务和运营无法信任剩余课时数据。

目标：

- 班次中的 `已上课时` 改为只读。
- 学习计划和班次的已上课时统一由排课记录驱动。

### P0-4. 学习计划后端仍开放直接增删改接口，破坏“由班次自动生成”的边界

当前后端保留了 `/plans` 的 `POST / PUT / DELETE` 能力，而前端虽然没开放按钮，但接口仍可直接调用。

风险：

- 可以绕开班次同步逻辑，手工改学习计划。
- 会出现“学习计划与班次、排课、权益不一致”的脏数据。
- 学习计划页边界被破坏，后续排查问题很难。

目标：

- 学习计划作为派生记录，不允许独立手工增删改。
- 后端接口要收口为只读，或只允许内部同步调用。

### P0-5. 课程产品改名后，学习计划课程名称可能滞后

学习计划里保存的是 `productName` 文本快照，课程产品改名后，学习计划不一定同步更新。

风险：

- 班次页显示新名字，学习计划页仍显示老名字。
- 前台、教练、运营对同一课程的叫法不一致。
- 续费和沟通时会出现口径混乱。

目标：

- 明确课程产品改名后的同步策略。
- 第一版至少保证班次和学习计划展示口径一致。

### P0-6. 页面字段不足以支撑真实的教学跟进和续费跟进

当前列表字段只有：

- 学员
- 手机号
- 班次
- 课程
- 教练
- 课时进度
- 状态

风险：

- 只能看“在不在班里”，看不出“最近有没有来上课”“还能不能继续约”“要不要马上续费跟进”。
- 管理员看完一行仍要跳多个页面查信息。
- 页面停留在台账层，不能服务日常运营动作。

目标：

- 增加少量高价值字段，不堆复杂信息。
- 让管理员在 1 行内完成基础判断，在 1 次详情展开内完成进一步理解。

### P0-7. 当前页面缺少明确动作出口，管理效率低

当前学习计划页没有直接去处理后续动作的入口。

风险：

- 看完后仍要手动切去学员页、班次页、排课页、权益页。
- 页面只“能看”，不能“推进工作”。

目标：

- 增加有限的轻操作。
- 只提供必要跳转，不把复杂编辑塞进学习计划页。

## 4. 从网球俱乐部管理员 / 运营视角，这个页面应该解决什么问题

真实俱乐部管理员、教务、运营打开学习计划页，核心不是看一张列表，而是要快速回答下面几个问题：

### 4.1 这个学员现在还在正常上课吗

需要快速判断：

- 所属班次是什么
- 跟哪个教练
- 最近一次什么时候上课
- 课时还剩多少

### 4.2 这个学员还能不能继续约课

需要快速判断：

- 班次进度还有多少
- 当前可用权益还有没有
- 权益什么时候到期

### 4.3 这个学员现在要不要做续费 / 转班 / 继续跟进

需要快速判断：

- 课程是否接近结课
- 最近有无持续上课
- 最近反馈情况如何

### 4.4 我下一步应该去哪处理

需要快速进入：

- 学员详情
- 班次详情
- 排课
- 权益查看

因此，学习计划页应承担“跟进中台”职责，但不承担复杂编辑职责。

## 5. 页面边界与联动边界

为避免学习计划页继续失控，必须明确边界。

### 5.1 学习计划页负责什么

核心职责：

- 以“学员 × 班次”为单位查看学习推进情况
- 帮助管理员判断是否需要继续排课、续费、跟进
- 作为学员教学状态的轻量中台

### 5.2 学习计划页不负责什么

不负责：

- 手工创建学习计划
- 手工编辑学习计划课时
- 直接修改班次核心信息
- 直接编辑权益规则
- 填写和编辑课后反馈

### 5.3 与班次页的边界

班次页负责：

- 建班
- 调整班次信息
- 决定学员属于哪个班次

学习计划页只读这些结果，不反向承担建班工作。

### 5.4 与排课页的边界

排课页负责：

- 具体某次上课的执行记录
- 消课
- 时间和资源冲突校验

学习计划页只引用“最近上课 / 下次排课 / 去排课”入口，不在本页直接编辑课程记录。

### 5.5 与权益账户页的边界

权益账户页负责：

- 真实课包余额
- 到期规则
- 适用校区 / 教练 / 时段

学习计划页只展示“权益摘要”，不承担权益核算和权益维护。

### 5.6 与学员页的边界

学员页负责：

- 学员基本资料
- 关联关系总览
- 历史反馈与基础信息编辑

学习计划页只保留“看学员”入口，不承接学员主档编辑。

## 6. 信息层级设计

本次改造必须把页面拆成 2 层，而不是继续堆在一个表格里。

### 6.1 第一层：列表页

作用：

- 找人
- 判断状态
- 快速筛选
- 快速进入下一步动作

这一层只放高价值字段，不做大段详情展示。

### 6.2 第二层：详情侧板 / 详情抽屉

作用：

- 快速看清这个学员在当前学习计划上的完整上下文
- 不跳出页面就理解这条记录

这一层负责“看”和“跳转”，不负责复杂编辑。

### 6.3 设计原则

必须遵守：

- 列表负责判断
- 详情负责理解
- 编辑归原页面处理

不允许把大量修改表单继续塞进学习计划页。

## 7. 当前字段是否足够满足需求

结论分两层。

### 7.1 满足基础台账查看：够

当前已有字段足以支撑：

- 知道学员在哪个班次
- 知道跟哪个教练
- 知道班次大致课时进度
- 知道记录状态

### 7.2 满足真实门店管理与运营动作：不够

当前最缺的不是更多静态字段，而是 4 类高价值信息：

1. 时间状态
2. 权益状态
3. 教学上下文
4. 动作出口

因此结论是：

- 当前字段足够做“学习计划台账”
- 不足以做“学习计划运营”

## 8. 90 分目标下的字段方案

本次改造不追求大而全，只补最有价值的字段。

### 8.1 列表层保留字段

- 学员
- 手机号
- 班次
- 课程
- 教练
- 班次进度
- 状态

### 8.2 列表层新增字段

建议新增：

1. `最近上课`
   - 用于判断学员是否持续上课
   - 比“状态”更贴近日常运营判断

2. `权益摘要`
   - 展示当前匹配的有效权益信息
   - 形式保持轻量，例如：`私教 10 节 · 剩 8/10 · 到期 2026-05-01`

3. `操作`
   - `详情`
   - `看学员`
   - `去排课`
   - `看班次`

### 8.3 详情层展示内容

建议详情抽屉包含 5 个区块：

1. `学习计划摘要`
   - 学员
   - 班次
   - 教练
   - 校区
   - 已上 / 总课时
   - 剩余课时
   - 状态

2. `最近排课`
   - 最近 3 次上课记录
   - 时间、场地、教练、课时

3. `下一步排课参考`
   - 下一次已排课
   - 若无，则仅显示“去排课”入口

4. `权益摘要`
   - 当前权益名称
   - 剩余课时
   - 到期日
   - 适用校区 / 教练简述

5. `最近反馈`
   - 最近 2 条课后反馈摘要
   - 仅展示时间、知识点 / 下次训练要点

### 8.4 本次明确不加的内容

第一版不做：

- 异常提示中心
- 自动打分
- 复杂流失模型
- 复杂 BI 图表
- 在学习计划页内直接编辑班次 / 权益 / 反馈

## 9. 页面结构方案

### 9.1 顶部说明区

保留“学习计划由班次自动生成”的说明，但文案要更准确：

- 说明这是一条“学员在某班次里的学习推进记录”
- 说明真实可约课以权益规则为准

### 9.2 顶部筛选区

建议保留或增加以下筛选：

- 搜索：学员 / 手机号 / 班次 / 订场关联账户
- 状态
- 校区
- 教练
- 课程 / 课程类型
- 课时阶段

其中“课时阶段”建议做轻量枚举：

- 全部
- 刚开课
- 进行中
- 临近结课

这里不是异常系统，只是管理筛选。

### 9.3 主列表字段顺序

建议顺序：

1. 学员
2. 手机号
3. 班次
4. 课程
5. 教练
6. 最近上课
7. 班次进度
8. 权益摘要
9. 状态
10. 操作

理由：

- 先看“是谁”
- 再看“在哪里上”
- 再看“最近有没有在上”
- 再看“剩多少”
- 最后决定“下一步做什么”

## 10. 页面交互方案

### 10.1 详情抽屉

每行点击 `详情` 后打开，不跳页面。

目标：

- 让运营在当前上下文内完成理解
- 避免在多个页面反复切换

### 10.2 轻操作按钮

每行只保留有限动作：

- `详情`
- `看学员`
- `去排课`
- `看班次`

说明：

- `看学员` 跳到学员页并打开对应学员
- `去排课` 打开排课弹窗，预选当前班次与学员
- `看班次` 跳到班次页并打开对应班次

### 10.3 不做原地编辑

学习计划页不新增：

- 新建学习计划按钮
- 编辑学习计划按钮
- 手工修正课时按钮

这些动作全部归原业务页。

## 11. 联动口径方案

### 11.1 班次 -> 学习计划

保留现有机制：

- 班次创建 / 编辑时自动同步学习计划

同时补充要求：

- 学习计划是班次派生记录，不允许绕开班次单独维护

### 11.2 排课 -> 班次 / 学习计划

保留现有机制：

- 排课保存 / 编辑 / 删除时，继续联动班次和学习计划课时

同时补充要求：

- 班次和学习计划中的 `usedLessons` 只由排课驱动

### 11.3 权益账户 -> 学习计划

新增展示口径：

- 学习计划展示班次进度
- 旁路展示权益摘要

明确：

- 学习计划页不是最终扣课核算页
- 真正能否继续约课，仍以权益账户规则为准

### 11.4 学员 / 教练改名 -> 学习计划

保留现有联动：

- 学员改名、改手机号，学习计划同步更新
- 教练改名，学习计划同步更新

### 11.5 课程产品改名 -> 班次 / 学习计划

新增要求：

- 课程产品改名后，班次与学习计划展示口径必须同步
- 不允许班次页和学习计划页各显示一套名称

## 12. 权限与数据边界方案

### 12.1 学习计划页前端权限

管理端：

- 管理员可见

教练端：

- 本次不扩展到教练端独立学习计划页

### 12.2 学习计划后端权限

原则：

- 学习计划记录不允许作为独立业务对象开放写接口

建议：

- `/plans` 仅保留 `GET`
- `POST / PUT / DELETE` 收口或内部化

### 12.3 数据真相来源

必须明确：

- 班次归属真相：班次表

## 13. 验收标准

改造完成后，学习计划页至少要满足：

1. 管理员能一眼看出某个学员在哪个班次、跟哪个教练、最近是否在正常上课。
2. 管理员能看到班次进度和权益摘要，不再只看旧课时余额。
3. 进度条表达与“课时进度”含义一致。
4. 学习计划不能再被独立手工增删改。
5. 班次页中的已上课时不再手工维护，学习计划进度由排课驱动。
6. 课程产品改名后，班次页和学习计划页展示口径一致。
7. 学习计划页可以直接进入学员、班次、排课等下一步动作。
8. 页面仍保持轻量，不演变成一个大而全的超级详情页。
