# 价格管理执行计划

> 给后续执行线程：按本文逐项执行。第一轮只做价格管理、自动带价、价格快照和渠道核销基础，不做完整财务系统，不做大重构。

**目标：** 把场地价格、会员折扣价、渠道商品价放进系统，由系统自动带价，运营只在特殊情况改价并留痕。

**架构：** 新增独立「价格管理」页面。场地价格和渠道商品价格统一维护；订场流水、会员储值扣款、渠道核销引用价格方案并保存价格快照。后续财务模块基于这些业务流水汇总。

**技术栈：** 原生 HTML/CSS/JS、Express API、TableStore、Node.js 断言测试。

5. 不重构全站文件结构。

## 三、数据口径

### 1. 价格类型

`venue_rate`：场地时段价。

字段：

| 字段 | 说明 |
|---|---|
| campus | 校区 |
| dateType | 工作日 / 周末节假日 |
| startTime | 开始时间 |
| endTime | 结束时间 |
| unitPrice | 原价/小时 |
| status | active / inactive |
| effectiveFrom | 生效日期 |
| effectiveTo | 失效日期，可空 |
| notes | 备注 |

`channel_product`：渠道商品价。

字段：

| 字段 | 说明 |
|---|---|
| channel | 门店 / 大众点评 / 抖音 / 小程序 |
| productName | 商品名称 |
| productType | 订场券 / 体验课 / 小班课 / 课包 |
| businessType | court / lesson / package |
| durationMinutes | 时长，订场券必填 |
| salePrice | 售价 |
| status | active / inactive |
| effectiveFrom | 生效日期 |
| effectiveTo | 失效日期，可空 |
| notes | 备注 |

### 2. 订场流水价格快照

订场用户 `history` 里的订场消费新增以下字段：

| 字段 | 说明 |
|---|---|
| priceMode | venue_rate / channel_product / manual |
| pricePlanId | 使用的价格方案ID |
| channel | 门店 / 大众点评 / 抖音 / 小程序 |
| channelOrderNo | 平台订单号 |
| redeemCode | 核销码 |
| systemAmount | 系统应收价 |
| finalAmount | 最终成交价 |
| priceOverridden | 是否人工改价 |
| overrideReason | 改价原因 |
| memberDiscount | 会员折扣，如 0.9 / 0.8 |

金额使用规则：

1. `systemAmount` 是系统根据价格方案算出来的金额。
2. `finalAmount` 是本单最终成交价。
3. `amount` 继续保留，等于 `finalAmount`，兼容现有财务流水。
4. 如果 `finalAmount !== systemAmount`，必须填写 `overrideReason`。

## 七、整体验收标准

最终上线前必须满足：

1. 管理端有独立「价格管理」页面。
2. 场地价格不再只靠本地表格维护。
3. 大众点评商品价格可以在系统里维护。
4. 普通订场能自动算价。
5. 会员储值扣款能按会员折扣自动算价。
6. 渠道商品能自动带价。
7. 运营可以改最终成交价。
8. 改价必须填写原因。
9. 每笔订场流水保存价格快照。
10. 历史流水不受后续价格修改影响。
11. `npm test` 通过，或明确说明和本需求无关的既有失败。
