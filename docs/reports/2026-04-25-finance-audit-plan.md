## 3.1 P0：财务中心顶部“订场收入 / 已入账”逻辑错误

现象：出现 `订场收入 < 已入账`。

已确认原因：

1. 页面没有完整识别 `financialLedger.actionType` 的全部枚举，尤其是 `核销`。
2. 页面统计“订场收入”时只算了部分动作。

影响：

1. 顶部卡片不可信。
2. 订场经营数据会误导决策。

## 6. 这次建议采用的财务质检机制

后续必须落地为正式审计机制，而不是临时脚本。

### 6.1 总账对账

同一天、同校区、同业务类型，比较：

1. 财务事实账本
2. 收入表
3. 消耗表
4. 页面顶部汇总

任一不一致即报错。

### 6.2 维度完整性检查

所有钱相关记录必须有：

1. `businessDate`
2. `actualCampusId`
3. `businessType`
4. `actionType`
5. `paymentChannel`
6. `sourceType`
7. `sourceId`

缺任何一个，都不允许进入正式经营报表。

### 6.3 枚举兜底检查

扫描全量：

1. `actionType`
2. `businessType`
3. `paymentChannel`
4. `ledgerType`

出现未定义枚举，直接列为 P0。

### 6.4 汇总一致性检查

必须同时满足：

1. 总数 = 各校区之和
2. 顶部卡片 = 明细表汇总
3. 同一指标跨页面只能有一个定义

### 13.1 教练端小程序：我的课表默认视图位置不对

你的原意我已经确认：

1. 不是改“默认定位到今天”这个逻辑
2. 只是改默认横向可视窗口的起点
3. 现在是一屏看到“昨天 / 今天 / 明天”
4. 你要的是一屏看到“今天 / 明天 / 后天”
5. 其他逻辑不要动

已确认原因：

1. 小程序课表横向滚动位置由 `timetableScrollLeft()` 计算。
2. 当前课表数据仍是严格的“本周 7 天视图”，由 `buildTimetableDays()` 生成本周 7 列。
3. 所以即使 `timetableScrollLeft()` 想把今天推到第一列，到了周六/周日也会被“本周只有 7 列”这个边界卡住。
4. 这就是为什么在 `2026-04-26` 周日，你仍然看到的是“周五 / 周六 / 周日”，而不是“今天 / 明天 / 后天”。
5. 代码位置主要在：
   - `wechat-miniprogram/miniprogram/utils/schedule.js`
   - `wechat-miniprogram/miniprogram/pages/schedule/schedule.js`

后续正确改法：

1. 不要只调 `scrollLeft`
2. 保留“数据仍按周”的前提
3. 单独生成一个“默认展示窗口”，固定取：
   - 今天
   - 明天
   - 后天
4. 周六/周日时，默认展示窗口允许跨到下周一 / 下周二
5. 不改“今天高亮”“本周切换”“回到本周”的业务含义

线程 4 当前进度：

1. `未完成`
2. 之前仅尝试改过默认滚动定位，但没有真正突破“本周 7 列”的边界限制
3. 当前正确方案应改为：“数据仍按周，默认窗口按今天起 3 天”

### 13.2 教练端小程序：首页“今日待办”的【查看详情】在部分手机被截断

已确认原因：

1. `今日排课` 卡片右侧按钮样式是 `.today-detail-btn`，宽度被写死为 `70px`。
2. 同时这张卡片右侧空间更紧，因为左边还有时间、课程标签、地点、学员等内容挤压。
3. `本周待办` 的按钮虽然也是 `70px`，但所在区域布局更宽、更稳定，所以不容易被截断。
4. 问题本质是：两个区域按钮文案一样，但承载它们的父布局不一样。

高概率直接根因：

1. `.today-detail-btn` 只有固定宽度，没有做更稳的文字容纳策略
2. 不同手机字号、渲染缩放或系统字体略大时，就会出现“查看详”

后续正确改法：

1. 只修 `today` 卡片按钮及其父容器布局
2. 不要顺手重做整页样式
3. 要同时验证 iPhone 小屏、安卓高字体缩放场景

2. 进入工作台后又会拉聚合接口 `/page-data/workbench`
3. 这个接口虽然比旧的 `/load-all` 轻，但仍会一次取：

### 13.4 网页端登录也慢

2. 体感慢主要发生在“登录后立刻加载当前页面数据”。
3. 当前页如果恰好是重页面，会马上触发聚合接口或多数据集后台加载。

结论：

1. 网页端慢的主因不是密码校验
2. 而是登录后首屏数据策略和当前落点页面过重

### 13.8 网页端管理端：每周循环排课报“上课时间不能跨天”

这个问题已经基本定位到代码级原因。

已确认原因：

1. 循环排课生成多周时间时，用了 `toISOString()`
2. `toISOString()` 会转成 UTC 时间
3. 对中国时区的早晨时间，转完后可能落到前一天
4. 再回到保存校验时，就会被判定成“开始和结束不在同一天”

所以像你说的“每周三 7-9 点”这种早晨时段，最容易中招。

这不是业务规则真的不允许，而是：

1. 循环时间生成用错了时间格式
2. 本地时间在批量生成时被错误转成 UTC 了

### 13.10 管理端排课日期在教练端“整体错一天”

这条这次**没有直接按你描述完整复现**，但已经把“能确认的”和“不能瞎下结论的”分开了。

已确认事实：

1. 当前线上 `ft_schedule` 真实数据里，现有排课记录的 `startTime/endTime` 都是本地格式字符串，例如：
   - `2026-04-24 16:00`
   - `2026-04-25 16:00`
   - `2026-04-26 13:00`
2. 当前线上 `ft_schedule` 共 `18` 条，本轮扫描结果：
   - `ISO/UTC 格式 0 条`
   - 以 `Z` 结尾的时间 `0 条`
   - 格式异常时间 `0 条`
   - `scheduleSource='循环排课'` 的记录 `0 条`
3. 我把你截图里相关学员的小程序周视图/课表分组逻辑直接跑了一遍，当前这些真实记录会被正确分到：

   - `2026-04-26`

已确认的系统性风险点：

2. 这会把本地时间转成 UTC。
3. 一旦后续再有循环排课、清晨时段、或任何 ISO 时间写入 `ft_schedule`，教练端网页、小程序、工作台这些按日期分组的页面就有可能出现错天。

3. 后续新线程必须先区分两类问题：
   - 标准本地时间排课是否错天
   - 循环排课 / ISO 时间排课是否错天

正确解决方向：

1. 排课存储层统一只存“本地业务时间字符串”，不要混 `toISOString()`
2. 教练网页、小程序、工作台全部统一使用同一套本地日期解析工具
3. 新线程先补一组时间口径测试：
   - 普通排课
   - 循环排课
   - 清晨时段
   - 周切换边界

### 13.11 学员购买 20 节课包时，保存不上或看起来保存了但没生效

这条已经查到一个**确定根因**，而且是系统性问题。

已确认原因：

1. 从 `学员信息` 页直接点【课包】打开购买弹窗时，弹窗里的“选择课包”直接依赖全局 `packages` 数据。

1. 课包购买入口的依赖数据策略是分裂的。

### 13.12 课包应该有归属校区，不能只靠“可用校区”

这条不是新需求空想，当前模型确实只做了一半。

已确认现状：

1. 现在售卖课包模型已经有 `campusIds`
2. 购买记录和权益账户也会复制 `campusIds`
3. 页面展示文案叫“可用校区”

但当前模型的根问题是：

1. `campusIds` 表达的是“可在哪些校区使用”
2. 它不等于“这笔课包销售归属哪个校区”
3. 也不等于“这个价格是按哪个校区定的”
4. 课包当前只有一个 `price`，没有真正的“定价校区 / 销售校区”字段

所以现在系统虽然能表达：

1. 这个课包可在哪些校区上课

但不能稳定表达：

1. 这个课包是按哪个校区定价卖出去的
2. 这笔购买应该算到哪个校区
3. 同名产品在不同校区不同价格时，财务和经营口径怎么落

正确解决方向：

1. 给课包补一个强语义字段，例如：
   - `saleCampusId` 或 `pricingCampusId`
2. 购买记录必须复制这个字段快照
3. 权益账户必须复制这个字段快照
4. 课包的“可用校区”和“销售归属校区”必须拆成两个不同维度

### 13.13 财务中心报错：`financialLedger is not defined`

这条当前代码库里没有直接复现出“同版本源码必炸”的情况，所以不能瞎写成“源码里少定义了变量”。

已确认事实：

1. 当前源码里，`financialLedger` 已在 `state.js` 顶层声明。
2. `financePage` 聚合接口也会回传 `financialLedger`。
3. 我本地用当前代码启动后，财务页可以打开，没有直接复现这条报错。

因此可以确定：

1. 这不是一个“当前这份源码必现”的静态代码错误。
2. 用户截图里的错误更像是**前端资源版本混用 / 缓存混用**导致的运行时问题。

已确认的系统性风险点：

1. `index.html` 里只有部分核心脚本带了版本参数。
2. 很多页面脚本仍然是裸 `src`，没有统一 cache busting。
3. 一旦页面脚本和状态脚本不是同一批版本，就可能出现：
   - 页面脚本引用了新变量
   - 但浏览器仍在跑旧状态脚本
   - 最终变成 `xxx is not defined`

当前最合理的根因判断：

1. 不是财务接口炸了
2. 是前端静态资源发布策略不一致，导致用户浏览器缓存了不兼容的脚本组合

正确解决方向：

1. 不要只补一个 `if(typeof financialLedger==='undefined')`
2. 必须统一整个前端静态资源的版本策略
3. 至少做到：
   - 所有核心页面脚本统一带版本参数或统一指纹
   - `index.html` 本身也纳入发布版本控制
   - 发布后避免旧 HTML 搭配新 JS，或旧 JS 搭配新 HTML
